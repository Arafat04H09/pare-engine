# Skill: Build

## Purpose
Implement specs from the Pare Engine spec library. Supports single-spec and batch modes. Reads specs, writes only owned files, self-tests with the verification command, and logs results.

## Inputs

Determine what to build based on the argument:

| Argument | Mode | Behavior |
|----------|------|----------|
| `specs/B-hardening/B2.1.md` | **Single** | Build one spec |
| `specs/B-hardening/` | **Folder** | Build all ready specs in that category folder |
| `--wave 1` | **Wave** | Build all ready specs in wave 1 (from `specs/index.md`) |
| `--next` | **Next** | Build the next ready spec (all deps completed) |
| `--next 3` | **Next N** | Build the next 3 ready specs |
| `--all` | **All** | Build all ready specs in dependency order |
| *(no argument)* | **Next** | Same as `--next` |

**"Ready" means:** Status is not "done" or "blocked", and all specs listed in its dependencies are "done".

### Resolving the Build Queue

For any batch mode (folder, wave, next N, all):

1. **Read `specs/index.md`** to get the spec list with statuses, waves, and dependencies.
2. **Filter to ready specs**: Exclude specs marked "done", "blocked", or "skipped". Exclude specs whose dependencies are not all "done".
3. **Order by wave, then by spec ID** within each wave (lower wave first, lower ID first within wave).
4. **For folder mode**: Further filter to only specs whose path starts with the provided folder.
5. The result is the **build queue** — an ordered list of specs to implement sequentially.

### Reading Context (for each spec)

Before implementing each spec, read:
- `CLAUDE.md` — Architectural constraints and coding conventions
- The `references/conventions.md` file in this directory — Quick reference for Pare coding rules
- All contract files referenced in the spec's READ-ONLY section
- **Build brief** (if it exists): Check `pipeline/5.5-prepare/brief-{spec-id}-*.md` for the pre-analyzed implementation guide. If a build brief exists, follow its recommended approach, patterns, and toolkit — it was generated by `/prepare` to make your work faster and more accurate.

## Process — Single Spec

For each spec in the build queue (or the single provided spec), execute these steps:

1. **Parse Spec**: Read the spec file. Extract:
   - Files OWNED (your exclusive write list)
   - Files READ-ONLY (import from, never modify)
   - Acceptance criteria (your checklist)
   - Verification command

2. **Read Context**: Read all FILES OWNED (if they exist) and FILES READ-ONLY to understand current state. Read relevant contract files to understand type interfaces.

3. **Implement**: Write code following the spec's technical approach:
   - Only create or modify files in the OWNED list
   - Import types from `@pare-engine/core/contracts`
   - Follow all coding conventions (named exports, async/await, Zod for LLM, etc.)
   - Use `Promise.allSettled()` for multi-provider calls
   - Use `generateObject()` with Zod schemas for structured LLM output

4. **Self-Test**: Run the verification command from the spec:
   ```bash
   pnpm build && pnpm test
   ```
   If it fails, read the error output and fix. Repeat until passing.

5. **Verify Boundaries**: Confirm you only modified files in your OWNED list:
   ```bash
   git diff --name-only
   ```
   If you modified a file outside your OWNED list, revert those changes.

6. **Log Result**: Append a build log entry to `pipeline/6-build/build-log-YYYY-MM-DD.md` with:
   - Spec ID and title
   - Files created/modified
   - Acceptance criteria status (pass/fail per criterion)
   - Verification command result
   - Any deviations from the spec (with justification)

7. **Update Index**: Mark the spec as "done" in `specs/index.md`.

## Process — Batch Orchestration (Parallel via Worktrees)

When building multiple specs, the orchestrator uses **git worktrees** for full isolation. Each sub-agent gets its own filesystem checkout so builds, tests, and file writes cannot interfere with each other.

### Phase 1: Setup

1. **Resolve Build Queue**: Follow "Resolving the Build Queue" above. Group specs by wave — specs within the same wave run in parallel; waves execute sequentially.

2. **Show Plan**: Display the queue to the user:
   ```
   Build Plan:
     Wave 1 (sequential — has dependencies):
       1. [A1] Build Fix
       2. [A2] Config Consolidation
     Wave 2 (parallel — 4 agents):
       3. [B1.1] Crawling
       4. [B1.2] Search
       5. [B2.1] Scoring Visibility
       6. [B2.2] Scoring Content
   ```

3. **Clean Stale Worktrees**: Remove any leftover worktrees from a previous failed run:
   ```bash
   # Remove stale worktrees
   git worktree prune
   rm -rf .wt/
   ```

### Phase 2: Execute Per Wave

For each wave in order:

**If wave has 1 spec** → run it directly in the main worktree (no isolation needed).

**If wave has 2+ specs** → run in parallel via worktrees:

#### 2a. Create Worktrees
For each spec in the wave:
```bash
git worktree add .wt/{spec-id} -b build/{spec-id} HEAD
```

#### 2b. Install Dependencies
Run in each worktree (fast — pnpm's content-addressable store means mostly symlinks):
```bash
cd .wt/{spec-id} && pnpm install --frozen-lockfile
```

#### 2c. Launch Parallel Sub-Agents
Launch one sub-agent per spec. Each sub-agent:
1. Operates entirely within `.wt/{spec-id}/`
2. Runs the single-spec process (steps 1-6 from "Process — Single Spec")
3. Does **NOT** update `specs/index.md` (orchestrator handles this)
4. Commits changes to its `build/{spec-id}` branch:
   ```bash
   cd .wt/{spec-id}
   git add -A
   git commit -m "build: {spec-id} — {title}"
   ```
5. Writes its build log to `.wt/{spec-id}/pipeline/6-build/build-log-{spec-id}.md`
6. Reports PASS or FAIL back to the orchestrator

**Sub-agents run concurrently.** Wait for all sub-agents in the wave to complete before proceeding.

#### 2d. Merge Results
For each spec that PASSED (in spec-ID order):
```bash
git merge build/{spec-id} --no-ff -m "build: merge {spec-id}"
```
Merges should be conflict-free because specs have non-overlapping file ownership. If a merge conflict occurs, it indicates a file ownership violation — log it and skip that spec.

For each spec that FAILED:
- Do NOT merge its branch
- Log the failure in the batch summary
- The branch is preserved for debugging (cleaned up in Phase 3)

#### 2e. Integration Verification
After merging all passed specs in a wave, run a single integration check:
```bash
pnpm build && pnpm test
```
If this fails, one of the merged specs broke something in combination. To identify which:
1. Reset to pre-merge state
2. Merge specs one at a time, testing after each
3. The first merge that breaks the build is the culprit
4. Mark that spec as FAIL, revert its merge, continue with the rest

#### 2f. Update Index
After successful integration, update `specs/index.md` for all passed specs in this wave. This is done by the orchestrator in the main worktree — sub-agents never touch the index.

#### 2g. Wave Cleanup
```bash
# Remove worktrees (use rm -rf + prune — git worktree remove can fail on Windows/OneDrive)
for spec in {spec-ids}; do
  rm -rf .wt/$spec 2>/dev/null
  git branch -d build/$spec 2>/dev/null
done
git worktree prune
```

Then proceed to the next wave (which may depend on specs from this wave).

### Phase 3: Finalize

1. **Consolidate Build Logs**: Gather per-spec build logs from worktrees (before cleanup) and append them to `pipeline/6-build/build-log-YYYY-MM-DD.md` in the main worktree.

2. **Write Batch Summary**: Write `pipeline/6-build/batch-YYYY-MM-DD.md` using the template at [templates/batch-summary.md](templates/batch-summary.md) containing:
   - Total specs attempted / passed / failed / skipped
   - Per-spec results table (with which wave each ran in)
   - Parallel execution info (how many agents ran per wave)
   - If any spec failed: error details and remaining queue
   - Integration verification results

3. **Final Cleanup**: Ensure no stale worktrees or branches remain:
   ```bash
   git worktree prune
   rm -rf .wt/
   ```

### Sequential Fallback

If worktree creation fails (e.g., disk space, path length limits on Windows), fall back to sequential execution in the main worktree. Log a warning: "Worktree creation failed — falling back to sequential mode."

In sequential mode, specs within a wave execute one at a time instead of in parallel. All other behavior (wave ordering, dependency checking, index updates, batch summary) remains the same.

## Output

- Code changes in OWNED files (for each completed spec)
- Per-spec log entries in `pipeline/6-build/build-log-YYYY-MM-DD.md`
- Batch summary in `pipeline/6-build/batch-YYYY-MM-DD.md` (batch mode only)
- Updated `specs/index.md` status for completed specs (mark as "done")
- Git history: one merge commit per passed spec (batch) or one commit per spec (sequential)

## Tooling

### File Ownership Enforcement (PreToolUse Hook)
A PreToolUse hook (`.claude/hooks/validate-file-ownership.sh`) blocks Edit/Write calls to files not in the spec's OWNED list. Sub-agents must have `PARE_SPEC_FILE` set to their spec path for enforcement to activate. When unset (normal dev), all writes are allowed.

### Merge Driver (pnpm-lock.yaml)
The `@pnpm/merge-driver` is installed and configured to auto-resolve `pnpm-lock.yaml` conflicts during branch merges. After merging worktree branches, run `pnpm install` to reconcile any remaining lockfile issues.

### MCP Servers
- **git-worktree** — Worktree creation, parallel workflow setup, cleanup
- **git** — Full git operations (27 tools) including merge, rebase, cherry-pick

## Rules
- **NEVER modify files outside your OWNED list.** This is the #1 rule. The PreToolUse hook enforces this — violations are blocked automatically when `PARE_SPEC_FILE` is set.
- **NEVER modify contract files** unless the spec explicitly owns them.
- Import shared types from `@pare-engine/core/contracts`, never from sibling code.
- Named exports only. No default exports (exception: Next.js pages).
- `async/await` only. No `.then()` chains.
- All environment variables come from validated config. No raw `process.env`.
- Use `generateObject()` + Zod for all LLM structured output.
- Use `Promise.allSettled()` for multi-provider queries. Never `Promise.all()`.
- Custom error classes extend `Error` with a `code` property.
- Sentiment analysis must be LLM-based, not keyword matching.
- HTML/CSS + Puppeteer for PDFs. Never React-PDF.
- If the spec seems wrong or impossible, log the issue and stop. Do not improvise.
- If implementation reveals that a vision assumption is wrong (e.g., an API doesn't work as expected, a feature is infeasible as designed), log it in the build log AND add a note to `VISION.md` or `PRODUCT_PLAN.md` with `<!-- Updated by build — YYYY-MM-DD -->`. Don't silently deviate — make the learning visible.
